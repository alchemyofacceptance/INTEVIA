## 1. Purpose of This Directory

This directory defines the Recognition Domain: the layer that transforms participation, contribution, learning, support, and evidence into governed identity movement.

Recognition is not certification.

Recognition is the organism reflecting evidenced becoming back to the human.

Recognition applies to:

* learners;
* developers;
* volunteers;
* early adopters;
* DISCUSSION contributors;
* external advocates;
* governance participants;
* field-signal contributors;
* capability-forming humans;
* v1.0 rollout contributors.

Recognition sits between Identity, Education, Events, Services, Library, Corpus, DISCUSSION, Governance Engine, CARE, and external advocacy surfaces.

This directory defines the public-safe architectural model for Recognition. It does not expose internal recognition engines, private modelling patterns, or governed practice materials.

## 3. v1.0 Boundary

Recognition is minimal but real in v1.0.

Included in v1.0:

* RecognitionRecord;
* RecognitionType;
* RecognitionEvidence;
* RecognitionState;
* identity linkage;
* education linkage;
* event linkage;
* service linkage;
* library linkage;
* DISCUSSION contribution recognition;
* early adopter / advocacy recognition;
* volunteer and developer contribution recognition;
* governance behaviour recognition;
* field-signal recognition;
* evidence and audit attachment points;
* governed review states;
* CARE boundary guidance.

Deferred unless ratified:

* badges;
* scoring;
* rankings;
* leaderboards;
* automated reputation systems;
* credential wallets;
* gamification;
* marketplace recognition;
* payment-linked recognition;
* automated recognition inference;
* certification engines.

Recognition must remain human-centred, evidence-based, non-coercive, and CARE-bounded.
