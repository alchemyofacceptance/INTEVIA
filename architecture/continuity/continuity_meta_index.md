# 4. Active vs Future Status

The active v0.1 Continuity Engine should distinguish files that exist and are governed from files that are future, optional, or planned.

Future or optional files should not expand the active implementation surface unless they are:

* created in the repository;
* reviewed for coherence;
* aligned with v1.0 boundaries;
* evidence-linked where appropriate;
* Human-governed and ratified where required.

This prevents the meta-layer from accidentally expanding continuity beyond the governed v1.0 surface.

---

# 5. Relationship to v1.0 Minimums

The Continuity Meta-Index may include the full conceptual architecture.

The v1.0 minimums define the smallest safe implementation substrate.

A file may be useful for navigation, orientation, review, or training without being required for v1.0 implementation.

This distinction protects implementation discipline.

The meta-index helps the organism see the whole architecture without requiring the whole architecture to become immediate build scope.
