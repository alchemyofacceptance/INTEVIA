# Continuity Archival v0.1

## Purpose

This file defines how continuity moves from active state into preserved lineage.

Archival is not abandonment.

Archival is not deletion.

Archival is not pausing.

Archival is the governed preservation of coherence when a thread is no longer active.

Continuity Archival answers:

> How is coherence preserved when a workstream, recognition, decision, or identity movement is no longer active?

Archival ensures that meaning, evidence, governance, and lineage remain intact without implying ongoing continuity.

---

## Archival Conditions v0.1

A continuity thread may be archived when:

* the workstream is complete;
* the workstream is intentionally closed;
* the workstream is superseded by a new governed thread;
* the Human retires the thread due to scope change;
* the thread cannot safely continue and must be preserved as lineage;
* the thread is no longer relevant to active coherence;
* the thread remains meaningful as evidence but should no longer drive active work;
* the thread has been reviewed and no longer requires re-entry support.

Archival must be Human-governed.

---

## Archival Requirements v0.1

An archival event should include:

* **timestamp** — when archival occurred;
* **archival_reason** — why the thread is no longer active;
* **prior_state** — the continuity state before archival;
* **evidence_links** — commits, posts, artefacts, corrections, decisions, screenshots, or other evidence objects;
* **continuity_markers** — signals that coherence was preserved up to archival;
* **linked_organs** — Recognition, Corpus, Governance Engine, Identity, or other relevant domains;
* **lineage_value** — what the archived thread preserves for future reference;
* **supersession_link** — any successor thread, artefact, decision, or version that replaces the archived thread;
* **human_interpretation_or_ratification** — Human-authored archival interpretation, correction, or ratification where applicable;
* **boundary_note** — any caution needed to prevent false continuity, overclaim, or hidden-state assumptions.

Archival preserves lineage, not active state.

---

## Archival Distinctions

### Archival vs Pausing

Pausing preserves active continuity for later re-entry.

Archival preserves lineage when active continuity is no longer required.

A paused thread may resume.

An archived thread should not resume unless explicitly reopened under Human governance.

### Archival vs Deletion

Deletion removes or discards material.

Archival preserves relevant meaning, evidence, governance, and lineage.

Archival is appropriate when the thread still has future interpretive or evidentiary value.

### Archival vs Supersession

Supersession occurs when a new governed thread, artefact, decision, or version replaces an earlier one.

Archival may preserve the superseded thread as lineage while pointing to the active successor.

The archived thread must not continue to govern active work unless explicitly restored.

### Archival vs False Continuity

Archival must not imply that a thread is still active.

If an archived thread is referenced later, it should be treated as lineage, not as live state.

---

## Archival Boundaries

Archival must not:

* imply the thread is still active;
* imply memory reconstruction;
* imply hidden state;
* preserve noise or drafts as lineage;
* convert incomplete work into continuity claims;
* override governance boundaries;
* conceal unresolved drift;
* avoid necessary deletion, correction, or supersession;
* preserve a thread merely because it exists;
* create false confidence about future recoverability.

Archival is a closure movement, not a continuation.
