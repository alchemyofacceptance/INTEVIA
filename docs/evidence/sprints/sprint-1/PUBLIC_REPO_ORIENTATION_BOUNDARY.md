# Public Repo Orientation Boundary

**Status:** Prepared boundary surface  
**Scope:** Public repo orientation only  
**Runtime impact:** None  
**Mutation authorised:** Boundary document only  
**Evidence-log change authorised:** No  

## 1. Purpose

This boundary defines the rails for a possible public repo orientation surface.

The purpose is to make the public INTEVIA repository easier for external readers to understand without adding runtime behaviour, product claims, maturity claims, or capability expansion.

This boundary exists because the public Sprint 1 trail now contains enough evidence to be meaningful, but not yet enough orientation to be easily understood by a new reader.

## 2. Candidate Orientation Surface

A later authorised Work Block may prepare one of the following public-legibility artefacts:

- a minimal `README.md` orientation pass;
- a `docs/START_HERE.md` public orientation note;
- a short "How to inspect Sprint 1" guide;
- a bounded repo navigation note.

No specific artefact is authorised by this boundary alone.

## 3. What the Orientation May Explain

A future orientation surface may explain:

- what INTEVIA is in ordinary public language;
- what Sprint 1 is currently doing;
- how to run the current visible surfaces;
- where the evidence trail lives;
- why governance and evidence appear alongside code;
- what HAT means at a high level;
- what OPC means as a bounded Sprint 1 working label;
- how to distinguish implementation files from evidence/governance files.

## 4. What the Orientation Must Not Claim

The orientation must not claim:

- product maturity;
- production readiness;
- architectural completion;
- scientific validation;
- general OPC validity;
- autonomous governance;
- autonomous recovery;
- AI authority;
- contributor readiness beyond what the repo supports;
- complete documentation;
- complete onboarding;
- external validation;
- ecosystem adoption.

## 5. Runtime / Capability Boundaries

The orientation must not introduce:

- `src` changes;
- new command modules;
- tests;
- routing;
- dynamic scanning;
- auto-discovery;
- runtime help behaviour;
- inference;
- hidden state;
- dependency changes;
- capability expansion.

## 6. Public-Language Boundary

The orientation should be clear enough for a new reader, but must remain evidence-bound.

It may say:

> INTEVIA is an early-stage open-source project exploring Human-governed AI-assisted organisational tooling through a public, evidence-first Sprint 1 build process.

It must not say:

> INTEVIA is a proven organismal programming system.

It may say:

> Sprint 1 currently exposes a minimal runnable surface, governance status, heartbeat command, inspect command, tests, evidence log, manifest, and boundary/candidate artefacts.

It must not say:

> Sprint 1 proves the Organismal Programming Construct.

## 7. Evidence Basis

The public repo arc currently shows:

`breathe -> status -> tests -> heartbeat -> inspect -> manifest -> evidence -> boundary -> candidate -> checkpoint`

Evidence-safe public interpretation:

> INTEVIA did not simply gain files. It gained visible governance sequence.

This supports a public orientation surface, but does not authorise claim expansion.

## 8. Breaker Conditions

The work must halt if a proposed orientation surface:

- claims maturity;
- claims proof;
- implies AI autonomy;
- implies runtime capability not present in the repo;
- hides Human governance;
- collapses HAT into generic AI tooling;
- treats OPC as proven doctrine;
- turns Sprint 1 evidence into marketing overclaim;
- mutates runtime files;
- expands command behaviour.

## 9. Expected Completion Condition

A public repo orientation boundary may be considered complete only when:

- the boundary is created;
- content is echoed and inspected;
- mutation scope is limited to this file;
- commit is created;
- push is confirmed;
- clean working tree is verified.

## 10. Keeper

> The public trail should become easier to understand before the system becomes more powerful.
