# START_HERE Boundary Charter

**Status:** Prepared boundary surface  
**Scope:** Public repo orientation candidate only  
**Candidate artefact:** `docs/START_HERE.md`  
**Runtime impact:** None  
**Mutation authorised:** Boundary document only  
**Evidence-log change authorised:** No  

## 1. Purpose

This boundary defines the rails for a possible `docs/START_HERE.md` public orientation surface.

The purpose is to help a new public reader understand the current INTEVIA Sprint 1 repo without adding runtime behaviour, onboarding claims, product maturity claims, architectural claims, or capability expansion.

This boundary exists because the public repo now has visible governance sequence, but still needs a clear, bounded orientation path.

## 2. Candidate Surface

If later authorised, Sprint 1 may prepare:

`docs/START_HERE.md`

This candidate may be a public-facing orientation note that explains:

- what INTEVIA is in ordinary public language;
- what Sprint 1 is currently doing;
- how to inspect the current minimal surfaces;
- where the evidence trail lives;
- why governance and evidence appear alongside code;
- what HAT means at a high level;
- what OPC means only as a bounded Sprint 1 working label;
- what is currently not claimed.

## 3. What START_HERE May Do

`docs/START_HERE.md` may:

- orient a new public reader;
- explain that INTEVIA is early-stage;
- explain that Sprint 1 is evidence-first;
- point to current visible surfaces such as `breathe`, `heartbeat`, `inspect`, tests, manifest, evidence log, and boundary/candidate artefacts;
- identify which files are implementation surfaces and which are evidence/governance surfaces;
- use plain language;
- remain bounded to observed Sprint 1 evidence.

## 4. What START_HERE Must Not Do

`docs/START_HERE.md` must not claim:

- product maturity;
- production readiness;
- architectural completion;
- scientific validation;
- general OPC validity;
- autonomous governance;
- autonomous recovery;
- AI authority;
- contributor readiness beyond what the repo supports;
- complete documentation;
- complete onboarding;
- external validation;
- ecosystem adoption.

## 5. Runtime / Capability Boundaries

The candidate must not introduce:

- `src` changes;
- new command modules;
- tests;
- routing;
- dynamic scanning;
- auto-discovery;
- runtime help behaviour;
- inference;
- hidden state;
- dependency changes;
- capability expansion.

## 6. Public-Language Boundary

The candidate may say:

> INTEVIA is an early-stage open-source project exploring Human-governed AI-assisted organisational tooling through a public, evidence-first Sprint 1 build process.

It may say:

> Sprint 1 currently exposes a minimal runnable surface, governance status, heartbeat command, inspect command, tests, evidence log, manifest, and boundary/candidate artefacts.

It must not say:

> INTEVIA is a proven organismal programming system.

It must not say:

> Sprint 1 proves the Organismal Programming Construct.

It must not say:

> INTEVIA is ready for production use.

## 7. Evidence Basis

The public repo arc currently shows:

`breathe -> status -> tests -> heartbeat -> inspect -> manifest -> evidence -> boundary -> candidate -> checkpoint -> orientation boundary`

Evidence-safe public interpretation:

> INTEVIA did not simply gain files. It gained visible governance sequence.

This supports preparing a `START_HERE.md` candidate, but does not authorise claim expansion.

## 8. Breaker Conditions

The work must halt if a proposed `START_HERE.md` surface:

- claims maturity;
- claims proof;
- implies AI autonomy;
- implies runtime capability not present in the repo;
- hides Human governance;
- collapses HAT into generic AI tooling;
- treats OPC as proven doctrine;
- turns Sprint 1 evidence into marketing overclaim;
- mutates runtime files;
- expands command behaviour;
- presents INTEVIA as production-ready;
- presents onboarding or contributor readiness beyond current evidence.

## 9. Expected Completion Condition

A `START_HERE.md` boundary charter may be considered complete only when:

- the boundary is created;
- content is echoed and inspected;
- mutation scope is limited to the authorised documentation/evidence file;
- commit is created;
- push is confirmed;
- clean working tree is verified.

## 10. Keeper

> The public trail should become easier to understand before the system becomes more powerful.
