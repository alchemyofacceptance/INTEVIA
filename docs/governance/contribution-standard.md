## 1. Purpose

This standard governs contributions to the INTEVIA repository.

It ensures:

- quality
- clarity
- structural integrity
- governance compliance
- maintainability
- Human qualification where required

Contributions must strengthen the organism rather than introduce drift.

---

## 2. Contribution Types

Contribution types include:

- code
- documentation
- governance artefacts
- visual assets
- diagrams
- architecture
- decisions
- specifications
- workflow maps
- narrative scaffolds
- Hidden Codex material where appropriately bounded

---

## 3. General Rules

All contributions must:

- use clear structure
- preserve repo readability
- avoid mixed formatting modes
- avoid unqualified Canonical changes
- maintain public/private boundaries
- respect the INTEVIA Markdown Governance Standard
- preserve Human-AI Triad governance

---

## 4. Pull Request Requirements

Pull requests should include:

- clear title
- meaningful description
- clean diffs
- relevant issue or context where applicable
- explicit note if governance, doctrine, or Canonical material is affected

Pull requests must not silently change Canonical material.

---

## 5. Commit Message Requirements

Commit messages should be:

- clear
- searchable
- scoped where appropriate
- technically meaningful
- aligned with the change actually made

Commit messages may include Hidden Codex flavour where appropriate, but the technical meaning must remain clear.

Example:

```text
feat(video-1): add INTEVIA 1440p v2 asset pack
```

The header is the technical index.  
The body may carry implementation meaning.  
Hidden Codex material must remain subordinate to repo clarity.

---

## 6. AI Contribution Rules

AI contributions must:

- output clean Markdown when producing repo text
- respect repository structure
- avoid doctrinal drift
- avoid accidental status promotion
- preserve Working vs Canonical distinctions
- await Human qualification before Canonical status
- avoid inventing source-of-truth claims
- flag ambiguity rather than hiding it

AI is a contributor, not an authority.

---

## 7. Human Qualification

No contribution becomes Canonical without explicit Human approval.

Human qualification is required for:

- governance changes
- doctrine changes
- role definitions
- public/private boundary decisions
- Canonical status changes
- supersession of existing Canonical material

---

## 8. Working vs Canonical

Working material may be useful, structured, and repo-worthy.

However:

- Working is not Canonical
- Working may evolve
- Working may be superseded
- Working may require further review

Canonical material is stable and may only change through explicit update or supersession.

---

## 9. Summary

Contributions must be:

- clear
- governed
- qualified
- maintainable
- aligned with the Human-AI Triad

The Human governs.  
The AI contributes.  
The repo preserves.

This is the way of the HAT.
